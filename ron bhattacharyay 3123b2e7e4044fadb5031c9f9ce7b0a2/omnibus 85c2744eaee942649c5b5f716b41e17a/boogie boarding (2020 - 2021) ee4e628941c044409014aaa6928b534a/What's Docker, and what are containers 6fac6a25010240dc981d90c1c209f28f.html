The TL;DR

Containers are a way to run code in isolated boxes within an operating system; they help avoid disastrous failures and add some consistency to how you set up applications.

  • Special software called a hypervisor allows multiple isolated operating systems to run on one computer, and containers do the same for operating systems (ü§Ø)

  • The most popular container engine is Docker: it‚Äôs open source and used by >30% of all developers

Social distancing, but for your bad code

To understand Docker, you need to understand virtual machines.

The first post I ever wrote for Technically was about the cloud, and how companies run their software on big, rented servers instead of building that infrastructure themselves. The biggest provider of that cloud, AWS, is on track to make more than $40B this year (!), and it‚Äôs all thanks to one technology: the hypervisor. The hypervisor allows AWS to run code from multiple different people on one server instead of giving everyone their own; it‚Äôs a transformational way of building things. But why is this important? The big hairy truth of software is that your code is going to fail, and it‚Äôs going to cause problems. Developer environments suck

Isolation isn‚Äôt the only thing that containers bring to the table. A problem that constantly plagues developers is the environment that their code runs in. Code has a lot of dependencies:

  • Libraries and frameworks
  • Environment variables
  • Language versions
  • Access to data and plugins
  • Networking between services

All of these (and more!) make up your environment. Every time you install or upgrade a new package, switch to a language version, or allow configure a connection to a database, you‚Äôre changing your environment.

The typical workflow for developers is to develop locally and then deploy to server. These problems are the worst, and they‚Äôre a constant when you‚Äôre building and deploying software. It‚Äôs like moving houses, but keeping all of your furniture. And the wallpaper.

Docker 101

Docker is an app like any other: you can download it from the Docker website, and you‚Äôll see a little icon with preferences and stuff. When you‚Äôre running Docker on a server though, it‚Äôs all through the command line. Let‚Äôs run through some of the basics:

  1. Docker images

The basic starting point for a Docker container is an image. An image is like a blueprint for what you want your containers to look like: once you‚Äôve built an image, you can create infinite containers from it that look exactly the same. A Docker image is just a static file, which makes it very sharable: Docker even has a product called Docker Hub where you can host and share your container images.

  1. The Dockerfile

The Dockerfile (aptly named) is how you actually build a Docker image. It‚Äôs just a text file with a series of commands that tell the operating system what to do when you‚Äôre building a container: install this version of this package, run this script, open this port, etc. It‚Äôs common practice to use someone else‚Äôs publicly available Docker image as a base and build from there (this is usually called ‚Äúpulling‚Äù an image).

  1. Orchestrating containers for scale - ü§© Kubernetes ü§©

The terminology around Docker and containers is a lot like virtual machines: you spin them up and spin them down. That‚Äôs because another major benefit of a containerized architecture is easier scale. If your application is growing and needs more resources, you can spin up new Docker containers easily. Docker takes care of networking them together and other low level utilities like that; but that‚Äôs just the start. This is where Kubernetes comes in.

Big apps (and today, for some reason, even small ones) can be made up of a bunch of Docker containers: one for the app, one for the database, one for adjacent services, and so on. Orchestrating this complex workflow and making sure the right containers are running at the right time (and that they can talk to each other) is a hard job, and that (in a nutshell) is what Kubernetes does. Docker also has Swarm, which takes on a similar, more tightly scoped version of this. More on that in a future post.

Learning more

This is just a basic intro to containers. If you‚Äôre looking to learn more, here are a few areas to dive deeper into:
